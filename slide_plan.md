**说什么**：
> 为什么选这两个算法？因为它们解决的是操作系统最核心的两个问题：
>
> **第一个问题：如何公平分配 CPU？**
> 简单轮转（RR）对所有进程一视同仁，但现实中有些进程优先级更高。
> MLFQ 用多级队列，但优先级调整靠经验，无法精确控制。
> **Stride 调度**用数学公式保证：优先级高的进程获得更多 CPU 时间，比例精确可控。
>
> **第二个问题：如何高效创建进程？**
> 最简单的 fork 是复制整个地址空间，但子进程可能马上 exec，之前的复制全白费。
> **COW 写时复制**：fork 时只复制页表（零拷贝），只有真正写入时才复制物理内存。
>
> 这两个算法都是 Linux 内核的真实设计模式，我们完整还原了。

### Slide 4：Stride — 核心公式（代码块 + 图示）

**画面设计**：
```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  [白底]                                                 │
│                                                         │
│  ─────────────────────────────────────────────────────  │
│  Stride 调度算法                                        │
│  ─────────────────────────────────────────────────────  │
│                                                         │
│  ┌─────────────────────────────────────────────────────┐
│  │  stride = BIG_STRIDE / (256 - priority)             │
│  │                                                     │
│  │  进程 A: priority=10                                │
│  │    stride = 1,000,000 / (256-10) = 4065             │
│  │    ↓ 小步长 → 被选中次数多 → 获得更多 CPU           │
│  │                                                     │
│  │  进程 B: priority=133                               │
│  │    stride = 1,000,000 / (256-133) = 8130            │
│  │    ↓ 大步长 → 被选中次数少 → 获得较少 CPU           │
│  └─────────────────────────────────────────────────────┘
│                                                         │
│  ┌─────────────────────────────────────────────────────┐
│  │  调度规则：每次选 pass_value 最小的进程运行          │
│  │  运行后：pass_value += stride                       │
│  └─────────────────────────────────────────────────────┘
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**HTML 艺术字代码**：
```html

  <span class="cmt"># 进程 A: 高优先级</span><br>
  A.priority = <span class="num">10</span><br>

</div>

<!-- 调度规则 -->
<div class="callout-v2">
  调度规则：每次选 <strong>pass_value 最小</strong>的进程运行<br>
  运行后：pass_value += stride
  <span class="cite">Scheduling Rule</span>
</div>
```

**说什么**：
> Stride 调度的核心公式：**stride = BIG_STRIDE / (256 - priority)**
>
> - priority 越小（优先级越高），stride 越小
> - stride 越小，pass_value 增长越慢
> - pass_value 增长越慢，被选中的次数越多
>
> 举个例子：
> - 进程 A 优先级 10，stride = 4065（小步长）
> - 进程 B 优先级 133，stride = 8130（大步长）
>
> A 的步长是 B 的一半，所以 A 被选中的次数是 B 的两倍。
> **这不是靠经验调优，是靠数学公式保证的公平性。**

```html
      • 浪费内存<br>
      • 子进程可能马上 exec，复制全白费
    </div>
  </div>
  <div class="card-e" style="border-top:3px solid #27AE60">
    <div class="card-num">✅</div>
    <div class="card-icon">⚡</div>
    <div class="t-cat" style="color:#27AE60">聪明办法 (COW)</div>
    <div style="font-weight:600;font-size:1.3vw">fork 时共享，写入时才复制</div>
    <div style="opacity:.6;margin-top:1vh">
      • 快<br>
      • 节省内存<br>
      • 只复制被修改的那一小块
    </div>
  </div>
</div>
```

**说什么**：
> 第二个核心算法：COW 写时复制。
>
> 传统的 fork 是「笨办法」：复制整个地址空间。但子进程可能马上 exec，之前的复制全白费。
>
> COW 是「聪明办法」：
> - fork 时：父子进程共享同一份物理内存，只复制页表（零拷贝）
> - 写入时：才真正复制被修改的那一小块
>
> 这就是 Linux 内核 fork 高效的关键。

## 第9段：性能对比（1.5分钟）

### Slide 14：COW vs 传统 Fork 对比

**画面设计**：
```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  [白底]                                                 │
│                                                         │
│  ─────────────────────────────────────────────────────  │
│  性能对比 — COW vs 传统 Fork                            │
│  ─────────────────────────────────────────────────────  │
│                                                         │
│  ┌─────────────────────────────────────────────────────┐
│  │  测试场景：fork 后子进程不写入任何数据               │
│  │                                                     │
│  │  传统 Fork:                                         │
│  │    fork 前空闲帧: 252                               │
│  │    fork 后空闲帧: 248                               │
│  │    额外开销: 4 帧 = 16KB                           │
│  │                                                     │
│  │  COW Fork (我们的实现):                             │
│  │    fork 前空闲帧: 252                               │
│  │    fork 后空闲帧: 252                               │
│  │    额外开销: 0 帧 = 0KB                            │
│  │                                                     │
│  │  内存节省: 100%                                     │
│  └─────────────────────────────────────────────────────┘
│                                                         │
│  ┌─────────────────────────────────────────────────────┐
│  │  结论：COW 在 fork 后不写入的场景下，               │
│  │        内存开销为零，节省 100%                      │
│  └─────────────────────────────────────────────────────┘
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**说什么**：
> 最后展示一下 COW 和传统 Fork 的性能对比。
>
> **传统 Fork**：fork 时复制整个地址空间。
> 如果子进程有 4 页内存，fork 后会多分配 4 帧，额外开销 16KB。
>
> **COW Fork（我们的实现）**：fork 时只复制页表，共享物理内存。
> fork 后空闲帧数不变，额外开销为 0。
>
> **内存节省 100%**。
>
> 这就是 COW 的价值：在 fork 后不写入的场景下，内存开销为零。
> 而现实中，很多进程 fork 后立即 exec，根本不会写入父进程的数据。
