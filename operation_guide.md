## ▌演示 1：Fork + ps + tree（~1.5min）

### 打开终端，启动程序

```bash
cd process-management
python main.py
```

**等待启动画面出现，看到 `[PID:0] os>` 提示符。**

### 敲第一条命令：fork

```
[PID:0] os> fork shell
```

**预期输出**：
```
子进程已创建: PID=1
```

**说**：
> fork 就像 Linux 的 fork 系统调用，创建一个子进程。
> 父进程拿到子进程的 PID=1。

### 敲第二条命令：再 fork 一个

```
[PID:0] os> fork editor
```

**预期输出**：
```
子进程已创建: PID=2
```

### 敲第三条命令：ps 查看进程列表

```
[PID:0] os> ps
```

**预期输出**：
```
PID    PPID   名称           状态         优先级    CPU时间
------------------------------------------------------------
0      -1     init           READY        255      0.0      *
1      0      shell          RUNNING      128      5.0
2      0      editor         READY        128      0.0

  总计: 3 个进程
```

**说**：
> ps 命令列出所有进程。可以看到 init 是根进程，shell 和 editor 都是它的子进程。
> 状态、优先级、CPU 时间都和 Linux 的 ps 命令对应。

### 敲第四条命令：tree 看进程树

```
[PID:0] os> tree
```

**预期输出**：
```
进程树:
└── PID=0
    ├── PID=1
    └── PID=2
```

**说**：
> 进程树清晰展示了父子关系。init 在根节点，两个子进程挂在下面。
> 这和 Linux 的 pstree 命令一样。

### 敲第五条命令：waitpid 回收

```
[PID:0] os> wait -1
```

**预期输出**：
```
子进程退出，exit_code=0
```

**说**：
> wait -1 表示等待任意子进程退出。
> 父进程回收了子进程的退出码，僵尸进程被清理。

### 再看一次进程列表

```
[PID:0] os> ps
```

**预期输出**：
```
PID    PPID   名称           状态         优先级    CPU时间
------------------------------------------------------------
0      -1     init           READY        255      0.0      *
1      0      shell          RUNNING      128      5.0

  总计: 2 个进程
```

**说**：
> editor 已经被回收，进程列表少了一个。
> 这就是完整的 fork → exec → waitpid 生命周期。

**要点**：每敲一条命令，停 1 秒让评委看到输出。边敲边解释，不要沉默操作。

## ▌回到 Slide 05 · COW 核心原理

**操作**：按 ← 回到 PPT，翻到 Slide 05。

**画面**：代码块 + HOW IT WORKS 三步骤

## ▌演示 2：COW 验证（~1.5min）

### 切回 VSCode

**操作**：按 Esc → overview → 点 VSCode。

### 创建子进程，查看内存

```
[PID:0] os> fork child
```

**预期输出**：
```
子进程已创建: PID=1
```

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 0
```

**说**：
> fork 之前，已使用 4 帧，共享帧为 0。

### 推进时钟，触发 COW

```
[PID:0] os> tick 5
```

**预期输出**：
```
已推进 5 个时钟周期
```

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 4
```

**说**：
> 注意看！已使用还是 4 帧，但共享帧从 0 变成了 4。
> 这说明父子进程共享了同一份物理内存，没有额外分配。
> 零拷贝，零额外开销。

**要点**：用手指着「共享帧: 4」这个数字，停 2 秒让评委看清楚。

### 触发写入，观察 COW 复制

```
[PID:0] os> write 4 "test cow"
```

**预期输出**：
```
写入 8 字节: test cow
```

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 5
  空闲: 251
  共享帧: 3
```

**说**：
> 写入之后，已使用从 4 变成 5，共享帧从 4 变成 3。
> 只复制了被修改的那一帧，其余三帧继续共享。
> 这就是 COW 的精髓：按需复制，节省内存。

## ▌Slide 07 · Stride 公平调度

**操作**：按 ← 回到 PPT，翻到 Slide 07。

**画面**：代码 + 柱状图 + 数学证明

**说**：
> 第二个创新点：Stride 公平调度。
> 左边是核心代码，右边是 200 个时间片的调度结果。
> A 的优先级是 10，B 和 C 的优先级是 133。

## ▌演示 3：Stride 公平性验证（~1.5min）

### 切回 VSCode

**操作**：按 Esc → overview → 点 VSCode。

### 创建三个进程，设置不同优先级

```
[PID:0] os> fork A
```

**预期输出**：
```
子进程已创建: PID=1
```

```
[PID:0] os> nice 1 10
```

**预期输出**：
```
进程 1 优先级已设为 10
```

**说**：
> A 的优先级设为 10，数值越小优先级越高。

```
[PID:0] os> fork B
```

**预期输出**：
```
子进程已创建: PID=2
```

```
[PID:0] os> nice 2 133
```

**预期输出**：
```
进程 2 优先级已设为 133
```

```
[PID:0] os> fork C
```

**预期输出**：
```
子进程已创建: PID=3
```

```
[PID:0] os> nice 3 133
```

**预期输出**：
```
进程 3 优先级已设为 133
```

**说**：
> B 和 C 的优先级都是 133，比 A 低。

### 推进 200 个时间片

```
[PID:0] os> tick 200
```

**预期输出**：
```
已推进 200 个时钟周期
```

### 查看结果

```
[PID:0] os> ps
```

**预期输出**：
```
PID    PPID   名称           状态         优先级    CPU时间
------------------------------------------------------------
0      -1     init           READY        255      0.0      *
1      0      A              RUNNING      10       99.4
2      0      B              READY        133      50.1
3      0      C              READY        133      50.3
```

**说**：
> 200 个时间片之后，A 的 CPU 时间约 100，B 和 C 各约 50。
> 比例是 2:1:1，和公式计算的结果完全一致。
> 这不是靠经验调优，是靠数学证明的公平性。

**要点**：数字可能略有偏差（±1），这是正常的，强调「接近 2:1:1」。

## ▌演示 4：管道 IPC（~1min）

### 继续在 VSCode 中操作

```
[PID:0] os> pipe
```

**预期输出**：
```
管道已创建: read_fd=3, write_fd=4
```

**说**：
> 创建一个管道，read 端是 fd=3，write 端是 fd=4。

```
[PID:0] os> fork child
```

**预期输出**：
```
子进程已创建: PID=4
```

```
[PID:0] os> write 4 "Hello from parent"
```

**预期输出**：
```
写入 18 字节: Hello from parent
```

**说**：
> 父进程往 fd=4 写入数据。

```
[PID:0] os> read 3
```

**预期输出**：
```
读取 18 字节: Hello from parent
```

**说**：
> 子进程从 fd=3 读出了同样的数据。
> 这就是 Linux 最基础的进程间通信方式——管道。
> 数据在内核缓冲区里传递，父子进程通过文件描述符读写。

**要点**：强调「父写子读」的方向性。

## ▌演示 6：级联退出 + 孤儿回收（~1min）

### 创建多级进程树

```
[PID:0] os> fork A
```

**预期输出**：
```
子进程已创建: PID=5
```

```
[PID:0] os> tree
```

**预期输出**：
```
进程树:
└── PID=0
    ├── PID=1
    ├── PID=2
    ├── PID=3
    ├── PID=4
    └── PID=5
```

**说**：
> 当前有 5 个子进程。

### 杀死一个进程，观察级联退出

```
[PID:0] os> kill 1
```

**预期输出**：
```
进程 1 已被杀死
```

```
[PID:0] os> tree
```

**说**：
> 杀死进程 1 后，它的子进程（如果有的话）会成为孤儿。
> 系统自动把孤儿重新挂到 init 下，这就是孤儿回收机制。
> 和 Linux 的 reparent_orphans 原理一致。

## ▌Slide 08 · 进程生命周期（回到 PPT）

**操作**：按 ← 回到 PPT，翻到 Slide 08。

**画面**：三栏 — 状态机 / 级联退出 / Fork Bomb 防护

**说**：
> 我们实现了完整的六状态机：NEW → READY → RUNNING → ZOMBIE → TERMINATED。
> 加上 SLEEPING 状态，所有转换都通过 set_state() 验证合法性。
> 级联退出三阶段：进程变 ZOMBIE → 子进程成孤儿 → 孤儿挂到 init。
> 还有 Fork Bomb 防护：最大进程数 64，max_children 64，init 不可杀死。

**要点**：手指向三个栏位，逐个点到。
