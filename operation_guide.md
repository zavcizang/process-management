## ▌演示 1：Fork + ps + tree（~1.5min）

### 第一步：启动程序

```bash

```

**等待启动画面出现，看到 `[PID:0] os>` 提示符。**

**说什么**：

> PID=0 是 init 进程，所有进程的根节点，相当于 Linux 的 1 号进程。

```
[PID:0] os> fork shell
```

**预期输出**：
```

**说什么**：
> fork 就像 Linux 的 fork 系统调用，创建一个子进程。
> 父进程拿到子进程的 PID=1。
> 子进程自动加入就绪队列，等待调度。

### 第三步：再 fork 一个

```
[PID:0] os> fork editor
```

**预期输出**：
```

```

**说什么**：
> 再创建一个子进程，名叫 editor，PID=2。
> 现在 init 下面有两个子进程了。

### 第四步：ps 查看进程列表

```
[PID:0] os> ps
```

**预期输出**：

## ▌回到 Slide 05 · COW 核心原理

**操作**：按 ← 回到 PPT，翻到 Slide 05。

**画面**：代码块 + HOW IT WORKS 三步骤

**说什么**：
> 接下来是我们的第一个创新点：COW 写时复制。
>
> 左边是核心代码，右边是原理。
> fork 的时候，子进程的页表指向同一个物理帧，ref_count 加 1。
> 只有当某一方写入的时候，才触发复制——这就是 Copy-On-Write。
>
> 具体来说：
> - fork_page_table() 函数：复制页表，设为只读，增加引用计数；
> - on_page_write() 函数：检查是否可写，如果不可写就复制物理帧，更新页表。
>
> 这和 Linux 内核的 copy_on_write() 函数设计一致。
> 接下来我们现场验证一下。

**要点**：
- 手指向代码的 `entry.is_writable = False` 和 `incref`

```

**预期输出**：
```

```

**说什么**：

### 第二步：查看 fork 前的内存状态

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 0
```

**说什么**：
> mem 命令显示内存状态。
> 总帧数 256，每帧 4KB，总内存 1MB。
> 已使用 4 帧，共享帧为 0——说明父子进程还没有共享内存。

### 第三步：推进时钟，触发 COW

```
[PID:0] os> tick 5
```

**预期输出**：
```
已推进 5 个时钟周期
```

**说什么**：
> tick 5 推进 5 个时钟周期，让调度器运行一段时间。

### 第四步：查看 COW 后的内存状态

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 4
```

**说什么**：
> 注意看！已使用还是 4 帧，但共享帧从 0 变成了 4。
> 这说明父子进程共享了同一份物理内存，没有额外分配。
> 零拷贝，零额外开销。
>
> 这就是 COW 的核心思想：fork 时只复制页表，不复制物理内存。
> 两个进程的页表指向同一个物理帧，引用计数加 1。
> 只有当某一方写入的时候，才复制那一帧——按需复制，节省内存。

**要点**：
- 用手指着「共享帧: 4」这个数字，停 2 秒让评委看清楚
- 这是关键证据，要说清楚

### 第五步：触发写入，观察 COW 复制

```
[PID:0] os> write 4 "test cow"
```

**预期输出**：
```
写入 8 字节: test cow
```

**说什么**：
> 现在我们触发一次写入，看看会发生什么。

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 5
  空闲: 251
  共享帧: 3
```

**说什么**：
> 写入之后，已使用从 4 变成 5，共享帧从 4 变成 3。
> 这说明：写入触发了 COW，复制了被修改的那一帧。
> 其余三帧继续共享，没有被复制。
>
> 总结一下：
> - fork 前：4 帧独占，0 帧共享；
> - fork 后：4 帧共享，0 帧独占——零拷贝；
> - 写入后：1 帧被复制，3 帧继续共享——按需复制。
>
> 这就是 COW 的精髓：fork 时零内存拷贝，写入时才复制。
> 好，接下来看第二个创新点：Stride 公平调度。

**要点**：
- 这是 COW 演示的高潮，要说清楚「从 4 变成 5，从 4 变成 3」
- 最后总结三行，让评委记住

## ▌Slide 07 · Stride 公平调度

**操作**：按 ← 回到 PPT，翻到 Slide 07。

**画面**：代码 + 柱状图 + 数学证明

**说什么**：
> 第二个创新点：Stride 公平调度。
>
> 左边是核心代码，右边是 200 个时间片的调度结果。
> 核心公式：stride = BIG_STRIDE / (256 - priority)
>
> 举个例子：
> - A 的优先级是 10，stride = 1,000,000 / (256-10) = 4065（小步长）
> - B 和 C 的优先级是 133，stride = 1,000,000 / (256-133) = 8130（大步长）
>
> 调度规则：每次选 pass_value 最小的进程运行，运行后 pass_value += stride。
> 步长小的进程，pass_value 增长得慢，所以被选中的次数多。
>
> 结果：A 跑了 49.7%，B 和 C 各跑了 25.1%，接近 2:1:1。
> 这不是靠经验调优，是靠数学证明的公平性。
>
> 接下来我们现场验证一下。

**要点**：
- 等柱状图动画播完再开口
- 手指向柱子，强调比例关系
- 「数学证明」四个字要说清楚

## ▌演示 4：管道 IPC（~1min）

### 继续在 VSCode 中操作

**说什么**：
> 接下来看管道 IPC——进程间通信。

### 第一步：创建管道

```
[PID:0] os> pipe
```

**预期输出**：
```
管道已创建: read_fd=3, write_fd=4
```

**说什么**：
> 创建一个管道，read 端是 fd=3，write 端是 fd=4。
> fd=0,1,2 保留给 stdin/stdout/stderr，管道从 fd=3 开始分配。
> 这和 Linux 的文件描述符分配规则一致。

### 第二步：创建子进程

```
[PID:0] os> fork child
```

**预期输出**：
```
子进程已创建: PID=4
```

**说什么**：
> 创建子进程，子进程继承了父进程的文件描述符表。
> 所以子进程也能访问同一个管道。

### 第三步：父进程写入

```
[PID:0] os> write 4 "Hello from parent"
```

**预期输出**：
```
写入 18 字节: Hello from parent
```

**说什么**：
> 父进程往 fd=4（写端）写入数据。
> 数据进入内核缓冲区，等待子进程读取。

### 第四步：子进程读取

```
[PID:0] os> read 3
```

**预期输出**：
```
读取 18 字节: Hello from parent
```

**说什么**：
> 子进程从 fd=3（读端）读出了同样的数据。
> 这就是管道 IPC：父写子读，数据在内核缓冲区里传递。
>
> 管道的特点：
> - 单向的——一个读端，一个写端；
> - 有缓冲区——4KB，满了阻塞写入，空了阻塞读取；
> - 基于文件描述符——fork 时继承，所以父子进程能共享。
>
> 这是 Linux 最基础的进程间通信方式。
> 好，接下来看内核日志。

**要点**：
- 强调「父写子读」的方向性
- 最后一句「接下来看内核日志」是过渡信号

### 一、关于 COW 写时复制

**Q1：COW 的触发时机是什么？是硬件支持还是软件实现？**

> 在真实的 Linux 内核中，COW 是靠硬件 MMU 的页错误中断（page fault）触发的。fork 时把页表项设为只读，写入时 CPU 触发缺页中断，内核在中断处理函数里判断是否需要复制。
>
> 在我们的模拟器中，因为是用户态 Python 实现，没有真正的硬件 MMU。我们通过在 `on_page_write()` 方法里检查 `is_writable` 标志来模拟这个过程。逻辑和 Linux 内核一致，只是触发方式从硬件中断变成了软件检查。

**Q2：引用计数怎么管理？会不会出现引用计数泄漏？**

> 每个物理帧有一个 `ref_count` 字段。fork 时 `incref`，exit 时 `decref`。当 `ref_count` 降到 0，帧被回收到空闲列表。
>
> 我们在 `ProcessManager.create_process()` 和 `exit_process()` 中都做了引用计数的维护。测试用例 `test_memory.py` 专门验证了 fork 后 ref_count 正确、exit 后帧被回收。

**Q3：如果父子进程同时写同一帧怎么办？**

> 在我们的实现中，调度是单线程的，不会出现真正的并发写入。但在设计上，每次写入前都会检查 `is_writable`，如果为 False 就触发 COW 复制。所以即使模拟并发，逻辑也是安全的。

**Q4：COW 的内存节省效果如何量化？**

> 从演示可以看到：fork 后共享帧从 0 变成 4，已使用帧数不变。只有写入时才多分配 1 帧。
>
> 如果进程有 N 页，传统 fork 需要复制 N 帧，COW 只在写入时按需复制。在实际场景中，很多进程 fork 后立即 exec，根本不会写入父进程的数据，所以 COW 节省了接近 100% 的内存拷贝。

### 二、关于 Stride 调度

**Q5：Stride 和 Linux CFS 的关系是什么？**

> Stride 调度是 CFS（Completely Fair Scheduler）的数学基础。CFS 使用虚拟运行时间（vruntime）来选择下一个运行的进程，vruntime 最小的先运行。这和 Stride 中选 pass_value 最小的进程运行是同一个思想。
>
> 我们的实现用最小堆维护 pass_value，每次调度选最小的，运行后加上 stride。时间复杂度 O(log n)，和 CFS 的红黑树 O(log n) 一样。

**Q6：为什么用最小堆而不是红黑树？**

> Python 标准库有 `heapq`，直接用最小堆实现更简洁。红黑树需要自己实现旋转、平衡等操作，代码量大且容易出错。在时间复杂度上两者都是 O(log n)，最小堆的常数因子还更小。
>
> 缺点是最小堆不支持高效删除任意元素，但我们用延迟删除（标记 + 跳过）解决了这个问题。

**Q7：pass_value 溢出怎么办？**

> 当所有进程的 pass_value 都超过 BIG_STRIDE 时，我们把所有进程的 pass_value 减去最小值，相当于重新从 0 开始。这不会影响公平性，因为减去的是同一个值，相对大小不变。
>
> 测试用例 `test_scheduler.py` 里有专门的溢出测试。

**Q8：优先级 0 和 255 分别代表什么？**

> 0 是最高优先级，255 是最低。stride = BIG_STRIDE / (256 - priority)，当 priority=0 时 stride 最小，跑得最多；当 priority=255 时 stride 最大，跑得最少。
>
> init 进程默认优先级 255，保证它不会抢占用户进程的 CPU 时间。

---

### 三、关于管道 IPC

**Q9：管道的缓冲区大小是多少？满了怎么办？**

> 缓冲区大小是 4KB（4096 字节），和 Linux 的默认管道缓冲区一致。
>
> 当缓冲区满时，写入操作会阻塞等待，直到有空间释放。当缓冲区空时，读取操作也会阻塞等待。这模拟了 Linux 管道的阻塞语义。

**Q10：管道是单向的还是双向的？**

> 单向的。fd=3 是读端，fd=4 是写端。父进程写 fd=4，子进程读 fd=3。
>
> 如果需要双向通信，需要创建两个管道。这和 Linux 的设计一致——管道天生就是单向的，避免了复杂的同步问题。

**Q11：文件描述符是怎么分配的？**

> fd=0,1,2 保留给 stdin/stdout/stderr，和 Linux 一样。管道从 fd=3 开始分配，每创建一个管道递增 2（一个读端、一个写端）。
>
> 子进程 fork 时继承父进程的文件描述符表，所以管道可以在父子进程之间共享。

### 四、关于进程生命周期

**Q12：什么是僵尸进程？为什么要回收？**

> 子进程 exit 后，它的退出码需要被父进程获取。在父进程调用 waitpid 之前，子进程处于 ZOMBIE 状态——已经死了，但还在进程表里占一个位置。
>
> 如果父进程一直不 wait，僵尸进程就会一直占用进程表项。在真实系统中，这会导致 PID 耗尽。所以我们的模拟器实现了 `waitpid` 来回收僵尸进程。

**Q13：孤儿进程怎么处理？**

> 当父进程先 exit，子进程还在运行，子进程就变成了孤儿。Linux 的做法是把孤儿进程重新挂到 init（PID=1）下面。
>
> 我们的实现用 `reparent_orphans()` 方法，在父进程 exit 时遍历其子进程，把它们的 ppid 改为 0（init）。测试用例 `test_kernel.py` 验证了这个行为。

**Q14：Fork Bomb 怎么防护？**

> Fork Bomb 是恶意进程不断 fork，耗尽系统资源。我们的防护措施：
> 1. 最大进程数 64——超过就拒绝 fork
> 2. 每个进程最大子进程数 64——防止单个进程创建过多子进程
> 3. init 不可杀死——保证系统不会因为根进程被杀而崩溃
>
> 这和 Linux 的 `ulimit -u`（最大进程数）原理一致。

### 七、刁钻问题

**Q21：你怎么证明你的调度是公平的？**

> 数学证明。stride_A / stride_B = (256-133) / (256-10) = 123/246 = 1:2。
> 所以 CPU 时间比应该是 2:1:1。
>
> 实际运行 200 个时间片，A 约 100，B 和 C 各约 50，比例接近 2:1:1。偏差来自取整和边界条件，但趋势是正确的。

**Q22：如果两个进程优先级相同，怎么调度？**

> stride 相同，pass_value 也相同的情况下，按 PID 顺序调度（先创建的先运行）。这和 Linux CFS 的行为一致——当 vruntime 相同时，选择先入队的进程。

**Q23：你的管道支持多个读者吗？**

> 目前不支持。我们的管道是 1:1 的，一个写端一个读端。Linux 的管道也是单读者的。
>
> 如果需要多读者，需要用共享内存 + 信号量，这是我们后续的扩展方向。

**Q24：init 进程能不能被 kill？**

> 不能。`kill 0` 会提示「不能杀死 init 进程」。这是 Fork Bomb 防护的一部分——如果 init 被杀，整个进程树就崩溃了。
>
> 在 Linux 中，init 也是不可杀死的（PID 1 有特殊保护）。

**Q25：你的模拟器支持多线程吗？**

> 不支持。我们的进程是独立的地址空间，不共享内存。线程是共享地址空间的执行流，需要额外的同步机制（互斥锁、信号量）。
>
> 这是我们后续可以扩展的方向，但目前的 scope 是进程管理，不是线程管理。
