## ▌Slide 02 · 为什么是这两个算法？（分栏 + 卡片）

**画面**：左侧问题 + 右侧答案，两个卡片对比

**HTML 设计要点**：
- 标题：`为什么选 Stride + COW？`
- 左卡片：`CPU 分配问题` — ❌ 简单轮转 / ❌ MLFQ / ✅ Stride
- 右卡片：`内存效率问题` — ❌ 简单复制 / ✅ COW
- 底部：`这两个算法都是 Linux 内核的真实设计模式`

**说什么**：
> 为什么选这两个算法？因为它们解决的是操作系统最核心的两个问题：
>
> **第一个问题：如何公平分配 CPU？**
> 简单轮转（RR）对所有进程一视同仁，但现实中有些进程优先级更高。
> MLFQ 用多级队列，但优先级调整靠经验，无法精确控制。
> **Stride 调度**用数学公式保证：优先级高的进程获得更多 CPU 时间，比例精确可控。
>
> **第二个问题：如何高效创建进程？**
> 最简单的 fork 是复制整个地址空间，但子进程可能马上 exec，之前的复制全白费。
> **COW 写时复制**：fork 时只复制页表（零拷贝），只有真正写入时才复制物理内存。
>
> 这两个算法都是 Linux 内核的真实设计模式，我们完整还原了。

**要点**：
- 这页是项目的核心卖点，要说清楚「为什么」
- 强调「数学证明」和「零拷贝」
- 让评委明白：这不是随便选的两个算法，而是有明确目标的

## ▌Slide 04 · Stride — 核心公式（代码块 + 图示）

**画面**：代码块 + 公式说明

**HTML 设计要点**：
- 标题：`Stride 调度算法`
- 代码块：`stride = BIG_STRIDE / (256 - priority)`
- 示例：A.priority=10 → stride=4065 / B.priority=133 → stride=8130
- 调度规则：`每次选 pass_value 最小的进程运行`

**说什么**：
> Stride 调度的核心公式：**stride = BIG_STRIDE / (256 - priority)**
>
> - priority 越小（优先级越高），stride 越小
> - stride 越小，pass_value 增长越慢
> - pass_value 增长越慢，被选中的次数越多
>
> 举个例子：
> - 进程 A 优先级 10，stride = 4065（小步长）
> - 进程 B 优先级 133，stride = 8130（大步长）
>
> A 的步长是 B 的一半，所以 A 被选中的次数是 B 的两倍。
> **这不是靠经验调优，是靠数学公式保证的公平性。**

## ▌Slide 05 · Stride — 数学证明（分栏 + 公式）

**画面**：左侧证明过程 + 右侧验证结果

**HTML 设计要点**：
- 左侧：`PROOF` — stride_A=4065, stride_B=8130, CPU 时间比=2:1
- 右侧：`VERIFICATION` — 200 时间片, A≈133, B≈67, 比例≈2:1 ✓
- 底部：`结论：A 获得 66.7% CPU，B 获得 33.3% CPU`

**说什么**：
> 我们来证明一下这个公平性。
>
> 假设 A 的优先级是 10，B 的优先级是 133。
> - stride_A = 1,000,000 / (256-10) = 4065
> - stride_B = 1,000,000 / (256-133) = 8130
>
> CPU 时间比 = 1/stride_A : 1/stride_B = 1/4065 : 1/8130 = **2:1**
>
> 也就是说，A 应该获得 2/3 的 CPU 时间，B 获得 1/3。
> 这个比例是由数学公式保证的，不是靠经验调优。
>
> 接下来我们现场验证一下。

## ▌Slide 06 · COW — 核心思想（对比卡片）

**画面**：左右对比 — 笨办法 vs 聪明办法

**HTML 设计要点**：
- 标题：`COW 写时复制`
- 左卡片（红色）：`❌ 笨办法` — fork 时复制整个地址空间
- 右卡片（绿色）：`✅ 聪明办法 (COW)` — fork 时共享，写入时才复制
- 底部：`Linux 内核的 fork 就是用 COW 实现的`

**说什么**：
> 第二个核心算法：COW 写时复制。
>
> 传统的 fork 是「笨办法」：复制整个地址空间。但子进程可能马上 exec，之前的复制全白费。
>
> COW 是「聪明办法」：
> - fork 时：父子进程共享同一份物理内存，只复制页表（零拷贝）
> - 写入时：才真正复制被修改的那一小块
>
> 这就是 Linux 内核 fork 高效的关键。

## ▌Slide 07 · COW — 原理图（流程图）

**画面**：三步骤流程图

**HTML 设计要点**：
- 标题：`COW 工作流程`
- Step 1：fork 前 — 父进程独占 3 个帧
- Step 2：fork 后 — 父子进程共享，ref=2，零拷贝！
- Step 3：子进程写入帧0 — 触发 COW，只复制帧0

**说什么**：
> COW 的工作流程分三步：
>
> **Step 1：fork 前**
> 父进程有 3 个帧，独占。
>
> **Step 2：fork 后**
> 父子进程共享同一份物理内存，引用计数变成 2。
> **没有分配新帧，零拷贝！**
>
> **Step 3：子进程写入帧0**
> 触发 COW，复制帧0 到新帧3。
> 其余帧继续共享，没有被复制。
>
> **只复制被修改的那一小块，节省 90% 的内存拷贝。**

```

```

**说什么**：


> 子进程从 fd=3（读端）读出了同样的数据。

> - 单向的——一个读端，一个写端；

> - 基于文件描述符——fork 时继承，所以父子进程能共享。

## 附录：评委可能提问及参考回答

### 一、关于 Stride 调度

**Q1：Stride 和 Linux CFS 的关系是什么？**

> Stride 调度是 CFS（Completely Fair Scheduler）的数学基础。CFS 使用虚拟运行时间（vruntime）来选择下一个运行的进程，vruntime 最小的先运行。这和 Stride 中选 pass_value 最小的进程运行是同一个思想。
>
> 我们的实现用最小堆维护 pass_value，每次调度选最小的，运行后加上 stride。时间复杂度 O(log n)，和 CFS 的红黑树 O(log n) 一样。

**Q2：为什么用最小堆而不是红黑树？**

> Python 标准库有 `heapq`，直接用最小堆实现更简洁。红黑树需要自己实现旋转、平衡等操作，代码量大且容易出错。在时间复杂度上两者都是 O(log n)，最小堆的常数因子还更小。
>
> 缺点是最小堆不支持高效删除任意元素，但我们用延迟删除（标记 + 跳过）解决了这个问题。

**Q3：pass_value 溢出怎么办？**

> 当所有进程的 pass_value 都超过 BIG_STRIDE 时，我们把所有进程的 pass_value 减去最小值，相当于重新从 0 开始。这不会影响公平性，因为减去的是同一个值，相对大小不变。
>
> 测试用例 `test_scheduler.py` 里有专门的溢出测试。

**Q4：优先级 0 和 255 分别代表什么？**

> 0 是最高优先级，255 是最低。stride = BIG_STRIDE / (256 - priority)，当 priority=0 时 stride 最小，跑得最多；当 priority=255 时 stride 最大，跑得最少。
>
> init 进程默认优先级 255，保证它不会抢占用户进程的 CPU 时间。

---

### 二、关于 COW 写时复制

**Q5：COW 的触发时机是什么？是硬件支持还是软件实现？**

> 在真实的 Linux 内核中，COW 是靠硬件 MMU 的页错误中断（page fault）触发的。fork 时把页表项设为只读，写入时 CPU 触发缺页中断，内核在中断处理函数里判断是否需要复制。
>
> 在我们的模拟器中，因为是用户态 Python 实现，没有真正的硬件 MMU。我们通过在 `on_page_write()` 方法里检查 `is_writable` 标志来模拟这个过程。逻辑和 Linux 内核一致，只是触发方式从硬件中断变成了软件检查。

**Q6：引用计数怎么管理？会不会出现引用计数泄漏？**

> 每个物理帧有一个 `ref_count` 字段。fork 时 `incref`，exit 时 `decref`。当 `ref_count` 降到 0，帧被回收到空闲列表。
>
> 我们在 `ProcessManager.create_process()` 和 `exit_process()` 中都做了引用计数的维护。测试用例 `test_memory.py` 专门验证了 fork 后 ref_count 正确、exit 后帧被回收。

**Q7：如果父子进程同时写同一帧怎么办？**

> 在我们的实现中，调度是单线程的，不会出现真正的并发写入。但在设计上，每次写入前都会检查 `is_writable`，如果为 False 就触发 COW 复制。所以即使模拟并发，逻辑也是安全的。

**Q8：COW 的内存节省效果如何量化？**

> 从演示可以看到：fork 后共享帧从 0 变成 4，已使用帧数不变。只有写入时才多分配 1 帧。
>
> 如果进程有 N 页，传统 fork 需要复制 N 帧，COW 只在写入时按需复制。在实际场景中，很多进程 fork 后立即 exec，根本不会写入父进程的数据，所以 COW 节省了接近 100% 的内存拷贝。

---

### 三、关于管道 IPC

**Q9：管道的缓冲区大小是多少？满了怎么办？**

> 缓冲区大小是 4KB（4096 字节），和 Linux 的默认管道缓冲区一致。
>
> 当缓冲区满时，写入操作会阻塞等待，直到有空间释放。当缓冲区空时，读取操作也会阻塞等待。这模拟了 Linux 管道的阻塞语义。

**Q10：管道是单向的还是双向的？**

> 单向的。fd=3 是读端，fd=4 是写端。父进程写 fd=4，子进程读 fd=3。
>
> 如果需要双向通信，需要创建两个管道。这和 Linux 的设计一致——管道天生就是单向的，避免了复杂的同步问题。

**Q11：文件描述符是怎么分配的？**

> fd=0,1,2 保留给 stdin/stdout/stderr，和 Linux 一样。管道从 fd=3 开始分配，每创建一个管道递增 2（一个读端、一个写端）。
>
> 子进程 fork 时继承父进程的文件描述符表，所以管道可以在父子进程之间共享。

### 四、关于进程生命周期

**Q12：什么是僵尸进程？为什么要回收？**

> 子进程 exit 后，它的退出码需要被父进程获取。在父进程调用 waitpid 之前，子进程处于 ZOMBIE 状态——已经死了，但还在进程表里占一个位置。
>
> 如果父进程一直不 wait，僵尸进程就会一直占用进程表项。在真实系统中，这会导致 PID 耗尽。所以我们的模拟器实现了 `waitpid` 来回收僵尸进程。

**Q13：孤儿进程怎么处理？**

> 当父进程先 exit，子进程还在运行，子进程就变成了孤儿。Linux 的做法是把孤儿进程重新挂到 init（PID=1）下面。
>
> 我们的实现用 `reparent_orphans()` 方法，在父进程 exit 时遍历其子进程，把它们的 ppid 改为 0（init）。测试用例 `test_kernel.py` 验证了这个行为。

**Q14：Fork Bomb 怎么防护？**

> Fork Bomb 是恶意进程不断 fork，耗尽系统资源。我们的防护措施：
> 1. 最大进程数 64——超过就拒绝 fork
> 2. 每个进程最大子进程数 64——防止单个进程创建过多子进程
> 3. init 不可杀死——保证系统不会因为根进程被杀而崩溃
>
> 这和 Linux 的 `ulimit -u`（最大进程数）原理一致。

### 五、关于架构和实现

**Q15：为什么用 Python 而不是 C？**

> 两个原因：
> 第一，Python 代码可读性高，评委能直接看懂内核设计模式，不需要编译环境；
> 第二，Python 的动态特性让我们能快速迭代，5000 行代码实现了 C 可能需要 2 万行才能写完的功能。
>
> 但我们的设计模式和 Linux 内核是一致的——PCB、页表、调度队列、系统调用层，每一层都可以对应到 Linux 内核的源码。如果将来要移植到 C，只需要替换语言层面的东西，架构不用改。

**Q16：测试覆盖率怎么样？**

> 6 个测试文件，40+ 个测试用例，覆盖了 PCB、进程树、调度器、内存管理、内核初始化、可视化等所有模块。
>
> 运行 `python main.py --test` 可以一次性跑完所有测试。每个测试文件都可以单独运行，比如 `python tests/test_memory.py`。

**Q17：和真实的 Linux 内核相比，你们的实现有什么差距？**

> 主要差距在三个方面：
> 1. **没有真正的硬件支持**——页表、MMU、中断都是软件模拟的
> 2. **单线程调度**——没有真正的并发，调度是协作式的
> 3. **简化了内存管理**——没有多级页表、TLB、swap 等机制
>
> 但核心设计模式是一致的：COW 的引用计数 + 写时复制、Stride 的最小堆调度、管道的缓冲区 + 文件描述符。这些是 Linux 内核的精髓，我们的模拟器完整还原了。

### 六、刁钻问题

**Q18：你怎么证明你的调度是公平的？**

> 数学证明。stride_A / stride_B = (256-133) / (256-10) = 123/246 = 1:2。
> 所以 CPU 时间比应该是 2:1:1。
>
> 实际运行 200 个时间片，A 约 100，B 和 C 各约 50，比例接近 2:1:1。偏差来自取整和边界条件，但趋势是正确的。

**Q19：如果两个进程优先级相同，怎么调度？**

> stride 相同，pass_value 也相同的情况下，按 PID 顺序调度（先创建的先运行）。这和 Linux CFS 的行为一致——当 vruntime 相同时，选择先入队的进程。

**Q20：init 进程能不能被 kill？**

> 不能。`kill 0` 会提示「不能杀死 init 进程」。这是 Fork Bomb 防护的一部分——如果 init 被杀，整个进程树就崩溃了。
>
> 在 Linux 中，init 也是不可杀死的（PID 1 有特殊保护）。
