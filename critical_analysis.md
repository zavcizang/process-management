> 本文档是对自己项目的冷静审视，不回避问题。

## 一、核心结论

**能拿奖，但拿不到最高奖。**

当前项目定位有一个**致命矛盾**：

> 你报的是「内核实现赛道」，但做的是「用户态模拟器」。

这是最大的风险。评委如果较真，会在这一点上扣分。

## 二、逐项分析

### 2.1 创新点评分

| 创新点 | 做到了什么 | 真正的创新度 | 评委视角 |
|--------|-----------|-------------|---------|
| COW 写时复制 | 引用计数 + 帧复制 | ⭐⭐⭐ 中等 | 能看出理解了原理，但**没有真正的内存管理**（没有页表、没有缺页中断、没有虚拟地址空间） |
| Stride 调度 | 比例公平性验证通过 | ⭐⭐⭐⭐ 较高 | 比 MLFQ 更有数学深度，评委可能没听过，**反而是加分项** |
| 级联退出 | 三阶段协议 + 孤儿回收 | ⭐⭐ 中等 | 实现正确，但这是教科书内容，不算创新 |
| 可视化 | ASCII + matplotlib 图表 | ⭐⭐ 中等 | 有演示效果，但不是算法创新 |

**真实创新度：1.5 个创新点（Stride 算法 + COW 的部分细节）**

### 2.2 代码质量评分

| 维度 | 得分 | 说明 |
|------|------|------|

### 2.3 致命短板

#### 短板 1：不是真正的内核实现

你的 fork 是 `PCB()` 对象创建，不是真实的进程创建。
你的 COW 是 `bytearray` 拷贝，不是真实的物理帧管理。
你的调度是数据结构操作，不是真实的 CPU 上下文切换。

**如果评委问「你的进程有真实的虚拟地址空间吗？」** — 答不上来。

#### 短板 2：没有"系统"的感觉

真正的内核实现会展示：
- 进程有自己的地址空间（页表映射）
- 真正的内存分配和回收
- 中断处理
- 进程间通信（IPC）

你的模拟器更像一个「进程管理教学工具」，不像一个「操作系统内核」。

#### 短板 3：演示场景不够惊艳

当前演示：
```
[PID:0] os> fork
子进程已创建: PID=1
[PID:0] os> ps
PID    PPID   名称           状态
0      -1     init           READY
1      0      shell          READY
```

这太平淡了。**没有可视化冲击力**。

## 四、改进 Plan（优先级排序）

### P0 — 必须做（不改会扣大分）

#### 4.1 补一个"内核级"功能

**方案 A：进程间通信（IPC）**

在现有架构上加一个简单的管道（Pipe）机制：

```python
# 新增文件
kernel/ipc.py      # 管道、消息队列
```

实现 `sys_pipe()`, `sys_write()`, `sys_read()`，让父子进程能通过管道通信。

**效果**：从"进程管理模拟器"升级为"进程管理 + IPC 模拟器"，更接近真实内核。

**方案 B：信号机制**

实现 `sys_signal()`, `sys_kill()`，支持 SIGTERM、SIGKILL 等信号。

**建议选方案 A（管道）**，因为：
- 实现难度适中（1-2天）
- 演示效果好（父写子读，直观）
- 评委认可度高（IPC 是 OS 核心概念）

#### 4.2 强化演示场景

当前演示太平淡。需要加入：

```
场景：父进程创建子进程，通过管道发送数据
$ python demos/pipe_demo.py

1. 创建管道 (fd=[3, 4])
2. fork 子进程 (PID=1)
3. 父进程写入: "Hello from parent"
4. 子进程读取: "Hello from parent"
5. 管道通信完成 ✓
```

### P1 — 应该做（提升竞争力）

#### 4.3 加一个内核日志系统

每次 fork/exit/wait 时记录日志，最后输出完整的操作序列：

```
[T=0]  CREATE pid=1 name=shell parent=0
[T=1]  FORK pid=1 → pid=2 name=editor
[T=2]  EXIT pid=2 exit_code=0
[T=2]  WAIT pid=1 waits_for=2 → exit_code=0
[T=3]  REAP pid=2 (zombie removed)
```

**好处**：
- 演示时能回放完整过程
- 证明你的实现是正确且完整的
- 类似 xv6 的 `dmesg` 输出

#### 4.4 加内存映射可视化

用 matplotlib 画一个「虚拟地址 → 物理帧」的映射图，展示 COW 前后的变化：

```
fork 前:                    fork 后:
┌───────┐                  ┌───────┐
│ 虚拟页0 │ → 帧0 (ref=1)   │ 虚拟页0 │ → 帧0 (ref=2) COW共享
│ 虚拟页1 │ → 帧1 (ref=1)   │ 虚拟页1 │ → 帧1 (ref=2) COW共享
└───────┘                  └───────┘

写入后:
┌───────┐
│ 虚拟页0 │ → 帧3 (ref=1) COW触发复制
│ 虚拟页1 │ → 帧1 (ref=2) 仍然共享
└───────┘
```

### P2 — 可以做（锦上添花）

#### 4.5 加 fork bomb 防护演示

```python
# demos/fork_bomb_demo.py
def fork_bomb(kernel):
    """演示 fork bomb 攻击和防护"""
    pid = kernel.create_process("attacker")
    count = 0
    while True:
        result = kernel.syscall.sys_fork(pid)
        count += 1
        if result < 0:
            print(f"  [防护] fork bomb 被拦截！尝试 {count} 次，成功 {count-1} 次")
            break
```
