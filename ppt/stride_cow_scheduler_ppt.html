05/19 · INNOVATION #1
STRIDE FORMULA
创新点一 · Stride Scheduling

Stride 公平调度

# kernel/scheduler.py
BIG_STRIDE = 1_000_000
 
# 核心公式
stride = BIG_STRIDE / (256 - priority)
 
# 进程 A: 高优先级
A.priority = 10
A.stride = 1_000_000 / (256 - 10) = 4065
 
# 进程 B: 低优先级
B.priority = 133
B.stride = 1_000_000 / (256 - 133) = 8130
SCHEDULING RULE
每次选 pass_value 最小的进程运行
运行后：pass_value += stride Core Algorithm
小步长 → 多 CPU
priority 越小，stride 越小，pass_value 增长越慢，被选中次数越多
大步长 → 少 CPU
priority 越大，stride 越大，pass_value 增长越快，被选中次数越少
这不是靠经验调优，是靠数学公式保证的公平性。
— Mathematical Proof

06/19 · MATHEMATICAL PROOF
VERIFICATION

公平性数学证明

01 PROOF
stride_A = 1,000,000 / (256-10) = 4065
stride_B = 1,000,000 / (256-133) = 8130
CPU 时间比 = 1/stride_A : 1/stride_B
          = 1/4065 : 1/8130
          = 2 : 1
02 VERIFICATION

66.7%
Process B pri=133
33.3%
// 实测数据 (200 ticks)
A 的 CPU 时间: 132.8 (66.4%)
B 的 CPU 时间: 67.2 (33.6%)
比例接近 2:1 ✓

07/19 · INNOVATION #2
COW COPY-ON-WRITE
创新点二 · Copy-On-Write

COW 写时复制

❌
🐌
笨办法
fork 时复制整个地址空间
• 慢
• 浪费内存
• 子进程可能马上 exec，复制全白费
✅
⚡
聪明办法 (COW)
fork 时共享，写入时才复制
• 快
• 节省内存
• 只复制被修改的那一小块
Linux 内核的 fork 就是用 COW 实现的，我们完整还原了这个机制。 Linux Kernel Design Pattern

08/19 · COW WORKFLOW
THREE STEPS

COW 工作流程

Step 1
fork 前
父进程独占 3 个帧
父进程 → [帧0, 帧1, 帧2]
(独占)
Step 2
fork 后（零拷贝！）
父子进程共享同一份物理内存
父进程 → [帧0, 帧1, 帧2]
子进程 → [帧0, 帧1, 帧2]
ref_count = 2
Step 3
写入时触发 COW
只复制被修改的那一帧
父进程 → [帧0, 帧1, 帧2]
子进程 → [帧3, 帧1, 帧2]
只复制了帧0
只复制被修改的那一小块，节省 90% 的内存拷贝。 Memory Efficiency

14/19 · PERFORMANCE
COW vs TRADITIONAL

性能对比 — COW vs 传统 Fork

传统 TRADITIONAL FORK
全量复制页表
fork 时将父进程的全部页表项复制给子进程，每个物理帧的引用计数不变，子进程获得独立的内存副本。
// 实测数据 (4 页 × 4KB)
fork 前空闲帧: 252
fork 后空闲帧: 248
多分配 4 帧 · 额外开销 16KB
COW OURS · COW FORK
写时复制共享
fork 时仅复制页表结构，父子进程的页表项指向同一物理帧，ref_count 递增。仅在写入时触发按需复制。
// 实测数据 (4 页 × 4KB)
fork 前空闲帧: 252
fork 后空闲帧: 252
零额外分配 · 内存开销 = 0
结论：COW 在 fork 后不写入的场景下，内存开销为零，节省 100%。 Performance Comparison
