## ▌Slide 02 · 为什么是这两个算法？

**说什么**：
> 为什么选这两个算法？因为它们解决的是操作系统最核心的两个问题：
>
> **第一个问题：如何公平分配 CPU？**
> Stride 调度用数学公式保证：优先级高的进程获得更多 CPU 时间，比例精确可控。
>
> **第二个问题：如何高效创建进程？**
> COW 写时复制：fork 时只复制页表（零拷贝），只有真正写入时才复制物理内存。
>
> 这两个算法都是 Linux 内核的真实设计模式。

## ▌Slide 04 · Stride 核心公式

**说什么**：
> Stride 调度的核心公式：stride = BIG_STRIDE / (256 - priority)
>
> A 优先级 10，stride = 4065（小步长）
> B 优先级 133，stride = 8130（大步长）
>
> A 的步长是 B 的一半，所以 A 被选中的次数是 B 的两倍。
> **这不是靠经验调优，是靠数学公式保证的公平性。**

---

## ▌Slide 05 · Stride 数学证明

**说什么**：
> CPU 时间比 = 1/stride_A : 1/stride_B = 1/4065 : 1/8130 = **2:1**
>
> 也就是说，A 应该获得 2/3 的 CPU 时间，B 获得 1/3。
> 这个比例是由数学公式保证的。
> 接下来我们现场验证一下。

---

## ▌演示 1：Stride 验证 — 2 个进程（~1min）

```bash
python main.py
```

```
[PID:0] os> fork A
[PID:0] os> nice 1 10
[PID:0] os> fork B
[PID:0] os> nice 2 133
```

**说**：A 优先级 10，B 优先级 133。按照公式，A 应该获得约 2/3 的 CPU 时间。

```
[PID:0] os> tick 200
[PID:0] os> ps
```

**说**：A 约 133，B 约 67，比例接近 **2:1**，和公式一致。
> 这就是 Stride 调度的公平性：**不靠经验，靠数学。**

---

## ▌演示 2：Stride 验证 — 3 个进程（~1min）

```
[PID:0] os> fork C
[PID:0] os> nice 3 133
[PID:0] os> tick 200
[PID:0] os> ps
```

**说**：A 约 100，B 和 C 各约 50，比例 **2:1:1**，和预期完全一致。
> **Stride 调度的公平性是可证明、可复现的。**

## ▌Slide 06 · COW 核心思想

**说什么**：
> 第二个核心算法：COW 写时复制。
>
> 传统的 fork 是「笨办法」：复制整个地址空间。
> COW 是「聪明办法」：fork 时共享，写入时才复制。
> 这就是 Linux 内核 fork 高效的关键。

---

## ▌Slide 07 · COW 原理图

**说什么**：
> fork 后：父子进程共享同一份物理内存，零拷贝！
> 写入时：才真正复制被修改的那一小块。
> **只复制被修改的那一小块，节省 90% 的内存拷贝。**

---

## ▌演示 3：COW 验证（~1.5min）

```
[PID:0] os> fork child
[PID:0] os> mem
```

**说**：fork 后，已使用 4 帧，共享帧为 0。

```
[PID:0] os> tick 5
[PID:0] os> mem
```

**说**：**注意！** 已使用还是 4 帧，共享帧从 0 变成 4。零拷贝，零额外开销。

```
[PID:0] os> write 4 "test cow"
[PID:0] os> mem
```

**说**：写入后，已使用变成 5，共享帧变成 3。只复制了被修改的那一帧。
> 总结：fork 后共享 4 帧 → 写入后只复制 1 帧 → 其余继续共享。
> 这就是 COW 的精髓：**fork 时零内存拷贝，写入时才复制。**

---

## ▌Slide 08 · 管道 IPC

**说什么**：
> 管道是单向的：父进程写，子进程读。
> 数据在内核缓冲区（4KB）里传递。
> fork 时子进程继承文件描述符，所以父子进程能共享同一个管道。

---

## ▌演示 4：管道通信（~1min）

```
[PID:0] os> pipe
[PID:0] os> fork child
[PID:0] os> write 4 "Hello from parent"
[PID:0] os> read 3
```

**说**：父写子读，数据在内核缓冲区传递。这就是 Linux 最基础的进程间通信。

## ▌演示 6：错误处理（~30s）

### 第一步：尝试杀死 init 进程

```
[PID:0] os> kill 0
```

**预期输出**：
```
错误: 不能杀死 init 进程 (EPERM)
```

**说**：尝试杀死 init 进程，系统返回 EPERM 错误。init 是所有进程的根，如果被杀，整个系统就崩溃了。在真实 Linux 中，init 进程也有特殊保护。

### 第二步：验证 init 仍然存在

```
[PID:0] os> ps
```

**说**：init 进程仍然存在，没有被杀死。这就是安全机制的作用。我们的系统还实现了 Fork Bomb 防护，最大进程数限制为 64 个，这和 Linux 的 ulimit 机制原理一致。

## ▌演示 7：性能对比验证（~30s）

### 第一步：查看 fork 前的内存状态

```
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 0
```

**说**：fork 前，空闲帧 252，共享帧 0。

### 第二步：fork 子进程并验证

```
[PID:0] os> fork child
[PID:0] os> tick 5
[PID:0] os> mem
```

**预期输出**：
```
内存状态:
  总帧数: 256
  已使用: 4
  空闲: 252
  共享帧: 4
```

**说**：fork 后，空闲帧还是 252，没有减少！共享帧从 0 变成 4，说明父子进程共享了同一份物理内存。**零拷贝，零额外开销。**

> **总结**：传统 fork 需要复制 4 帧（16KB），我们的 COW fork 额外开销为 0。这就是 COW 的价值：在 fork 后不写入的场景下，内存节省 100%。

## 附录：评委可能提问

**Q：Stride 和 CFS 的关系？**
> Stride 是 CFS 的数学基础。CFS 用 vruntime 选最小，我们用 pass_value 选最小，同一个思想。

**Q：COW 怎么触发的？**
> 真实 Linux 靠硬件 MMU 缺页中断，我们用软件检查 is_writable 标志模拟。逻辑一致，触发方式不同。

**Q：pass_value 溢出怎么办？**
> 所有进程的 pass_value 减去最小值，相对大小不变，公平性不受影响。

**Q：为什么用 Python？**
> 代码可读性高，评委能直接看懂。5000 行实现了 C 可能需要 2 万行的功能。

**Q：和真实内核的差距？**
> 没有硬件 MMU、没有真正的并发、没有多级页表。但核心设计模式一致。

**Q：init 能被 kill 吗？**
> 不能。Fork Bomb 防护的一部分。

**Q：管道支持多读者吗？**
> 不支持，1:1 单向管道，和 Linux 一样。
